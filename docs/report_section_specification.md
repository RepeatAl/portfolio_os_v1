# PORTFOLIO OS — REPORT SECTION SPECIFICATION
Version: v1
Status: Canonical Report Rendering SSOT

---

# PURPOSE

This document defines the canonical structure of Portfolio OS reports.

The report system must remain:

- deterministic
- explainable
- modular
- multilingual-ready
- structurally consistent

Reports are NOT freeform narratives.

Reports are structured PM reasoning artifacts.

This framework defines:

- mandatory report sections
- section ordering
- rendering rules
- explanation requirements
- section responsibilities
- language constraints

The goal is:

stable institutional-grade report generation.

---

# CORE PRINCIPLE

Reports must explain structure.

Reports must NEVER become:

- AI essays
- emotional narratives
- macro storytelling
- filler text generation
- generic market commentary

Every report section must answer:

"What structurally matters and why?"

---

# REPORT CHAIN

Portfolio OS follows this report structure:

Signals
→ Semantic States
→ PM Reasoning
→ Action Space
→ Language Rendering
→ Report Sections

No report section may bypass PM reasoning.

---

# REPORT OBJECTIVE

The report should answer:

- What environment currently exists?
- What structurally drives the portfolio?
- What creates fragility?
- What improves resilience?
- What dependencies matter?
- What options exist?
- What tradeoffs exist?

The report exists to improve understanding.

---

# CANONICAL REPORT ORDER

Portfolio OS uses the following canonical report order.

---

# SECTION 1 — EXECUTIVE SUMMARY

Purpose:
Provide immediate structural overview.

The section should explain:

- portfolio posture
- dominant structural condition
- biggest fragility
- biggest strength
- deployment posture
- overall structural interpretation

Length:

- short
- highly readable
- maximum clarity

The summary must NEVER become:

- emotional
- generic
- vague

---

# SECTION 2 — MARKET REGIME

Purpose:
Explain current market environment.

The section should explain:

- liquidity conditions
- participation quality
- volatility structure
- leadership concentration
- regime stability

The section must ALWAYS explain:

WHY the regime exists.

Bad Example:

"Neutral regime."

Good Example:

"Liquidity conditions remain stable,
while market participation stays selective
and leadership remains concentrated in AI-related growth exposure."

---

# SECTION 3 — PORTFOLIO STRUCTURE

Purpose:
Explain portfolio architecture.

The section should explain:

- allocation structure
- dominant exposures
- concentration
- thematic balance
- deployment structure

Questions answered:

- What structurally drives the portfolio?
- Which exposures dominate?

---

# SECTION 4 — CONCENTRATION & DEPENDENCY ANALYSIS

Purpose:
Explain hidden overlap and fragility.

The section should explain:

- thematic overlap
- narrative dependency
- correlated exposure
- hidden concentration
- structural amplification

Questions answered:

- What dependencies dominate?
- Which fragilities amplify downside?

The section must clearly separate:

company quality
from
portfolio dependency.

---

# SECTION 5 — DEPLOYMENT ANALYSIS

Purpose:
Explain capital positioning quality.

The section should explain:

- cash flexibility
- optionality
- deployment pacing
- exposure rigidity
- adaptability

Questions answered:

- How flexible is the portfolio?
- How resilient is deployment?

---

# SECTION 6 — SCENARIO ANALYSIS

Purpose:
Explain structural vulnerability.

The section should explain:

- stress sensitivity
- regime vulnerability
- downside amplification
- resilience under deterioration

The section must answer:

"What structurally changes if conditions worsen?"

The section must NEVER become:

- prediction theater
- dramatic forecasting

---

# SECTION 7 — ACTION SPACE

Purpose:
Present explainable portfolio options.

The section should explain:

- structural options
- diversification paths
- deployment alternatives
- resilience improvements
- tradeoffs

The section must NEVER generate:

- commands
- emotional urgency
- certainty framing

The system generates:

explainable options.

---

# SECTION 8 — CONFIDENCE INTERPRETATION

Purpose:
Explain reasoning stability.

The section should explain:

- why confidence exists
- which signals align
- which conditions conflict
- what ambiguity exists

Bad Example:

"Confidence: 82"

Good Example:

"Multiple structural signals currently align,
which increases the stability of the current interpretation framework."

Confidence must NEVER appear magical.

---

# SECTION 9 — PM SUMMARY

Purpose:
Provide final structural synthesis.

The section should explain:

- dominant structural reality
- primary fragility
- resilience state
- deployment posture
- reasoning conclusion

The PM Summary should feel:

- calm
- explainable
- institutional
- structurally grounded

NOT:

- emotional
- sensational
- predictive

---

# REPORT SECTION RULES

Every section must:

1. Explain meaning
2. Explain causes
3. Explain implications
4. Explain tradeoffs
5. Remain understandable
6. Remain deterministic
7. Remain semantically grounded

---

# REPORT LENGTH GOVERNANCE

Reports should remain:

- concise
- dense with meaning
- low on filler
- high on structural clarity

The system should avoid:

- repetition
- generic filler
- narrative loops
- verbose macro commentary

Meaning density dominates report length.

---

# MANDATORY EXPLANATION RULE

The report must NEVER present:

- unexplained scores
- unexplained regimes
- unexplained fragility
- unexplained confidence
- unexplained actions

Every important conclusion must explain:

- why it exists
- what caused it
- why it matters

---

# HUMAN READABILITY RULE

Reports must remain understandable by:

- non-professional investors
- advanced PM users
- long-term allocators
- strategic investors

The system must avoid:

- excessive institutional jargon
- macro-TV language
- pseudo-intelligent phrasing
- AI filler language

Understanding dominates sophistication.

---

# LANGUAGE GOVERNANCE RULE

Language rendering contains NO business logic.

Business logic already exists in:

- signals
- semantics
- PM reasoning
- action-space orchestration

Rendering only translates meaning.

This guarantees:

- multilingual consistency
- deterministic interpretation
- semantic stability

---

# MULTILINGUAL RENDERING RULE

Portfolio OS should support:

- English
- German
- future multilingual rendering

using identical semantic meaning.

The system translates:

meaning,
NOT logic.

---

# CONFLICT HANDLING RULE

If signals conflict:

- confidence decreases
- ambiguity increases
- action-space widens
- deterministic conviction weakens

Reports must openly acknowledge ambiguity.

The system must NEVER simulate certainty.

---

# REPORT TRACEABILITY RULE

Every major report conclusion should remain traceable to:

- signal origin
- semantic states
- engine outputs
- PM reasoning chain

No report conclusion may become disconnected from structural reality.

---

# REPORT CONSISTENCY RULE

The same semantic state must produce:

consistent reasoning behavior.

Example:

ai_dependency_high

must NOT produce:

- bullish interpretation in one report
- defensive interpretation in another

without supporting structural changes.

---

# REPORT FAILURE RULE

If important data is missing:

- reports degrade gracefully
- ambiguity becomes visible
- confidence weakens
- unsupported reasoning stops

The system must NEVER fabricate narrative continuity.

---

# DASHBOARD RELATIONSHIP

The dashboard visualizes report structure.

Examples:

- regime state
- concentration maps
- deployment flexibility
- dependency overlap
- scenario fragility
- action-space simulations

The dashboard visualizes reasoning outputs,
not independent truth.

---

# MEMORY RELATIONSHIP

Portfolio memory should eventually preserve:

- historical report structures
- prior PM summaries
- historical fragility
- deployment evolution
- recurring dependencies
- historical confidence states

The system should understand:

how portfolio reasoning evolves over time.

---

# LONG-TERM REPORT EVOLUTION

Future report systems may include:

- adaptive report density
- longitudinal PM reasoning
- scenario persistence analysis
- portfolio evolution summaries
- behavioral PM overlays
- historical fragility comparison
- semantic trend interpretation

All future report systems must remain:

- explainable
- deterministic
- semantically grounded
- structurally interpretable

---

# REPORT PHILOSOPHY

Portfolio OS reports do not exist to impress users.

They exist to:

- explain structural reality
- clarify portfolio conditions
- expose hidden fragility
- improve PM understanding
- reduce ambiguity

Reports are reasoning artifacts.

Not entertainment products.
