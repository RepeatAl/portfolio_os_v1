# PORTFOLIO OS — ACTION SPACE FRAMEWORK
Version: v1
Status: Canonical Action Space SSOT

---

# PURPOSE

This document defines how Portfolio OS generates action spaces.

Portfolio OS does NOT generate:

- direct trading commands
- forced buy/sell signals
- emotional urgency
- prediction-based execution advice

Portfolio OS generates:

- explainable portfolio options
- structural alternatives
- PM reasoning pathways
- tradeoff-aware portfolio adjustments

The system exists to support human portfolio decisions.

Not to replace them.

---

# CORE PRINCIPLE

Portfolio OS does not say:

"You must do this."

Portfolio OS says:

"These are the structurally explainable options,
their implications,
and their tradeoffs."

The system expands understanding.

The human decides.

---

# ACTION SPACE CHAIN

Portfolio OS follows this structure:

Portfolio State
→ Semantic States
→ PM Reasoning
→ Structural Priorities
→ Action Space Generation
→ Human Interpretation

Action spaces emerge from structure,
not prediction.

---

# ACTION SPACE OBJECTIVE

The system should answer:

- What can improve resilience?
- What reduces dependency?
- What improves diversification?
- What preserves flexibility?
- What increases fragility?
- What structural tradeoffs exist?
- What deployment paths exist?

The system explains possibilities,
not certainty.

---

# ACTION SPACE TYPES

Portfolio OS supports multiple action-space classes.

---

# TYPE 1 — RISK REDUCTION ACTIONS

Purpose:
Reduce structural fragility.

Examples:

- reducing concentration
- reducing dependency overlap
- lowering correlated exposure
- reducing deployment rigidity

Questions answered:

- What reduces downside sensitivity?
- What improves resilience?
- What lowers dependency concentration?

---

# TYPE 2 — DIVERSIFICATION ACTIONS

Purpose:
Improve structural balance.

Examples:

- adding healthcare exposure
- broadening geographic spread
- reducing thematic clustering
- balancing macro sensitivity

Questions answered:

- What broadens participation?
- What weakens narrative dependency?
- What improves structural flexibility?

---

# TYPE 3 — DEPLOYMENT ACTIONS

Purpose:
Improve capital efficiency.

Examples:

- staged deployment
- preserving optionality
- maintaining cash flexibility
- scaling exposure gradually

Questions answered:

- How much capital should remain flexible?
- What deployment pacing improves resilience?
- What deployment structures reduce fragility?

---

# TYPE 4 — REGIME-ADAPTIVE ACTIONS

Purpose:
Align portfolio posture with market environment.

Examples:

- defensive rotation
- selective growth exposure
- balanced regime participation
- liquidity-sensitive positioning

Questions answered:

- Which structures fit current conditions?
- Which exposures lose support?
- Which structures improve adaptability?

---

# TYPE 5 — MAINTENANCE ACTIONS

Purpose:
Preserve structural stability.

Examples:

- avoiding overexpansion
- maintaining diversification
- preserving deployment discipline
- avoiding narrative overcrowding

Questions answered:

- What should NOT change?
- What currently works structurally?
- What preserves resilience?

---

# ACTION SPACE STRUCTURE

Every action option should contain:

1. Structural Objective
2. Rationale
3. Benefits
4. Tradeoffs
5. Scenario Implications
6. Dependency Changes
7. Confidence Context

Action spaces must remain explainable.

---

# ACTION SPACE EXAMPLES

Bad Example:

"Sell semiconductor exposure now."

Good Example:

"Reducing semiconductor concentration may improve diversification
and reduce dependency on continued AI leadership,
but may also reduce participation in current growth momentum."

---

Bad Example:

"Deploy cash aggressively."

Good Example:

"Gradual deployment may improve participation
while preserving flexibility if market conditions weaken."

---

# TRADEOFF PRINCIPLE

Every action creates tradeoffs.

Portfolio OS must ALWAYS explain:

- what improves
- what weakens
- what flexibility changes
- what dependency changes
- what resilience changes

No action may appear consequence-free.

---

# ACTION SPACE PRIORITY MODEL

The system prioritizes:

1. Structural resilience
2. Diversification quality
3. Dependency reduction
4. Deployment flexibility
5. Scenario survivability
6. Regime compatibility
7. Tactical participation

Structural quality dominates emotional momentum.

---

# NO COMMAND RULE

Portfolio OS must NEVER generate:

- "must buy"
- "must sell"
- panic framing
- emotional urgency
- prediction certainty
- authority simulation

The system generates:

- options
- reasoning
- structural implications

NOT commands.

---

# CONFIDENCE RELATIONSHIP

Action-space confidence depends on:

- signal alignment
- structural coherence
- scenario consistency
- semantic stability
- regime clarity

Low confidence should:

- widen option ranges
- acknowledge ambiguity
- reduce deterministic framing

High confidence may:

- strengthen prioritization
- narrow structural focus
- clarify tradeoffs

---

# HUMAN READABILITY RULE

Action spaces must remain understandable.

The system must explain:

- why an action exists
- what structurally changes
- what tradeoffs emerge
- what flexibility changes
- what risks improve or worsen

The system must avoid:

- PM jargon
- vague strategic language
- emotional wording
- AI filler text

Understanding dominates sophistication.

---

# ACTION SPACE VS PREDICTION

Portfolio OS separates:

Action Quality
from
Prediction Certainty.

Example:

An action may structurally improve diversification
without guaranteeing short-term outperformance.

The system evaluates:

structural consequences,
not future certainty.

---

# REPORT RELATIONSHIP

Reports explain action-space meaning.

Reports should answer:

- why this option exists
- what improves
- what weakens
- what dependencies shift
- what structural implications emerge

Reports interpret consequences,
not commands.

---

# DASHBOARD RELATIONSHIP

Dashboards visualize:

- structural options
- diversification paths
- deployment pathways
- concentration reduction
- resilience improvements
- tradeoff structures

Dashboards visualize explainable choices,
not emotional signals.

---

# SIMULATION RELATIONSHIP

Every action should ideally support simulation.

The system should visualize:

BEFORE:
Current portfolio posture.

AFTER:
Structural changes after adjustment.

The simulation framework explains:

- dependency changes
- resilience shifts
- concentration reduction
- deployment evolution

Action spaces become more valuable when structurally visualized.

---

# MEMORY RELATIONSHIP

Portfolio memory should eventually track:

- repeated action patterns
- deployment discipline
- recurring fragility
- historical tradeoffs
- resilience evolution

The system should learn:

how portfolio behavior evolves over time.

---

# LONG-TERM ACTION SPACE EVOLUTION

Future systems may include:

- adaptive action sequencing
- dynamic deployment paths
- structural optimization pathways
- multi-step PM workflows
- scenario-weighted action maps
- dependency-aware rebalancing
- resilience optimization frameworks

All future systems must remain:

- explainable
- deterministic
- signal-based
- structurally interpretable

---

# ACTION SPACE PHILOSOPHY

Portfolio OS does not attempt to control users.

Portfolio OS attempts to:

- clarify structural options
- explain tradeoffs
- reduce hidden fragility
- improve deployment quality
- support PM reasoning

The system expands understanding.

The human remains the allocator.
