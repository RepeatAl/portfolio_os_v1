# PORTFOLIO OS — SYSTEM ARCHITECTURE
Version: v1
Status: Active SSOT

---

# PURPOSE

Portfolio OS is not a trading dashboard.

It is an explainable portfolio reasoning system.

The system exists to:

- interpret portfolio state
- interpret market state
- detect concentration and dependency risks
- explain portfolio exposure
- synthesize institutional-style signals
- generate explainable portfolio intelligence reports
- support future portfolio decisions

The dashboard is only a visualization layer.

The reasoning system is the primary product.

---

# CORE PRINCIPLE

The system must never generate narrative first.

The system must:

1. collect signals
2. interpret signals
3. generate semantic meaning
4. perform PM reasoning
5. render language output

Language is rendering.

Semantics are truth.

---

# SYSTEM LAYERS

The architecture is divided into 4 layers.

---

# LAYER 1 — RAW SIGNAL ENGINES

Purpose:
Generate measurable portfolio and market signals.

These engines produce:

- numerical data
- market relationships
- portfolio metrics
- flows
- volatility
- breadth
- concentration
- correlations
- scenario outputs

These engines DO NOT interpret meaning.

Examples:

- allocation_engine
- breadth_engine
- liquidity_engine
- flow_engine
- cross_asset_engine
- correlation_engine
- scenario_engine
- attribution_engine

Output Type:

Raw signals only.

---

# LAYER 2 — SEMANTIC INTERPRETATION

Purpose:
Translate raw signals into semantic portfolio states.

Examples:

- supportive_liquidity
- concentration_risk_elevated
- ai_dependency_high
- market_breadth_strong
- european_risk_on
- volatility_stable
- participation_broad

This layer is language-independent.

This is the canonical reasoning layer.

The semantic layer represents the official system truth.

---

# LAYER 3 — PM REASONING ENGINE

Purpose:
Explain what the semantic states actually mean.

This layer performs:

- portfolio interpretation
- risk explanation
- tradeoff analysis
- dependency explanation
- action-space generation
- scenario reasoning
- institutional PM-style synthesis

Examples:

- why concentration matters
- why current regime matters
- what happens under stress scenarios
- what the portfolio currently depends on
- what options exist

This layer must avoid generic AI language.

Reasoning must always remain:

- explainable
- signal-based
- evidence-based
- non-alarmist
- PM-oriented

---

# LAYER 4 — REPORT RENDERING

Purpose:
Render PM reasoning into human-readable language.

Examples:

- English report
- German report
- future multilingual rendering

Important:

Language rendering must NEVER contain business logic.

Business logic belongs to:

- semantic interpretation
- PM reasoning layer

Language is presentation only.

---

# DASHBOARD ROLE

The dashboard is NOT the product.

The dashboard visualizes:

- portfolio state
- market state
- PM reasoning
- scenarios
- simulations
- reports

The reasoning system remains primary.

---

# PORTFOLIO OS IDENTITY

Portfolio OS is:

- an explainable PM reasoning system
- a portfolio intelligence assistant
- a portfolio interpretation engine
- a signal orchestration platform

Portfolio OS is NOT:

- a hype-trading tool
- a prediction engine
- a signal spam dashboard
- a black-box AI stock picker

# SIGNAL ORIGINATION ARCHITECTURE

Portfolio OS is a signal-driven system.

The system does NOT generate opinions first.

All portfolio reasoning must originate from measurable signals.

---

# SIGNAL SOURCES

Signals may originate from:

## Portfolio Data

Examples:

- positions
- allocation
- exposure
- concentration
- market value
- P/L
- capital deployment
- watchlist structure

---

## Market Data

Examples:

- indices
- yields
- volatility
- breadth
- liquidity
- sector rotation
- relative strength
- macro proxies

---

## Derived Signals

Derived signals are generated by engines.

Examples:

- concentration risk
- AI dependency
- regime stability
- cross-asset divergence
- participation quality
- liquidity deterioration
- momentum exhaustion

Derived signals are NOT direct market data.

They are interpreted structural conditions.

---

# SIGNAL COLLECTION FLOW

The signal flow always follows this structure:

Data Source
→ Raw Engine Calculation
→ Derived Signal
→ Semantic Interpretation
→ PM Reasoning
→ Report Rendering

No reasoning may bypass this chain.

---

# ENGINE RESPONSIBILITIES

Each engine has exactly one responsibility.

Examples:

Allocation Engine:
Portfolio structure and exposure.

Regime Engine:
Market regime classification.

Breadth Engine:
Participation quality.

Liquidity Engine:
Liquidity conditions.

Correlation Engine:
Dependency relationships.

Scenario Engine:
Portfolio stress interpretation.

Attribution Engine:
Performance and concentration drivers.

No engine should generate narrative directly.

Narrative belongs only to the reasoning layer.

---

# SIGNAL VALIDATION

Signals should be cross-confirmed whenever possible.

Examples:

High concentration risk may be confirmed by:

- allocation engine
- correlation engine
- narrative dependency engine
- scenario engine

Confidence increases when multiple engines align.

---

# SIGNAL HIERARCHY

Not all signals have equal importance.

Priority order:

1. Structural portfolio risk
2. Liquidity conditions
3. Cross-asset confirmation
4. Breadth confirmation
5. Narrative concentration
6. Tactical momentum
7. Short-term volatility

Structural signals always dominate short-term signals.

---

# REASONING CONSTRAINT

The system must NEVER generate conclusions without signal support.

Every PM interpretation must be traceable to:

- signal origin
- engine source
- semantic interpretation

Explainability is mandatory.
---
# ARCHITECTURE RULES

1. Semantics before language.
2. Reports explain signals.
3. No generic AI filler language.
4. Every conclusion must be explainable.
5. Portfolio risk != asset quality.
6. The reasoning layer is the product.
7. Dashboards are visualization layers only.
