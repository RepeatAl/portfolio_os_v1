# Domainization Runtime Boundary Analysis — Domainization-Governance-Hardening

## Document Purpose

Analysis of domain ownership, boundary enforcement, and runtime mutation controls.
All findings are evidence-backed with file references. No implementation recommendations.

## 3. Runtime Mutation Analysis

### Can Artifacts Be Mutated Outside Their Lifecycle State?

**Answer: Yes.** There is no runtime gate preventing mutation.

| Lifecycle Constraint | Declared In | Enforced? | Evidence |
|---------------------|-------------|-----------|----------|
| `modifiable_states` | `lifecycle_state_machine.yaml` | No | `LifecycleValidator` in validation orchestrator warns but doesn't prevent |
| `read_only_states` | `lifecycle_state_machine.yaml` | No | Same — warning only |
| `regenerable_states` | `lifecycle_state_machine.yaml` | No | No code checks this before engine regeneration |

### Mutation Scenarios Without Governance

1. **Archived artifact modification**: An artifact in `archived` state (read-only per state machine) can be freely modified on disk — no pre-write check exists.

2. **Deprecated artifact regeneration**: An artifact in `deprecated` state can be regenerated by an engine — `regenerable_states` is not checked before write.

3. **Cross-domain write**: A SIGNALS domain artifact can be modified by code in the REPORT domain — `allowed_writers` is never checked.

4. **Lifecycle state bypass**: An artifact can jump from `draft` directly to `deprecated` even if the state machine doesn't define that transition — transitions are validated post-hoc by the observer, not prevented pre-write.

### Mutation Tracking Absence

| What's Tracked | How | Persistence |
|----------------|-----|-------------|
| `last_modified` field | Manual update in registry YAML | Persisted but unreliable (human-maintained) |
| Git diff of registry | Standard git history | Persisted but not structured for governance queries |
| Who changed what | Not tracked | **Absent** |
| Under what lifecycle state | Not tracked | **Absent** |
| Whether transition was valid | Logged during health report only | **Not persisted** |

## 4. Forbidden Interaction Analysis

### Detected Forbidden Interactions

Source: `engines/pipeline_orchestrator.py` method `detect_forbidden_flows()`

| Interaction Type | Detection Method | Prevention? |
|------------------|-----------------|-------------|
| Signal → Report (bypassing SEMANTICS + REASONING) | `detect_forbidden_flows()` checks coverage per category | **No** — logs `Severity.WARNING` only |
| Legacy engine_runner direct outputs | `_emit_briefing_deprecation_warnings()` checks for `.txt` briefing files | **No** — emits `DeprecationWarning` only |

### Currently Detected Forbidden Flows

From `pipeline_orchestrator.py` analysis:
- 13 of 14 signal categories lack semantic coverage (only `allocation` is covered)
- Each produces a forbidden flow warning: `{category}_engine → {section} (skipped: [SEMANTICS, REASONING])`
- The legacy `run_report_engine(regime, decision, quality)` call is itself a forbidden flow

### Undetected Forbidden Interactions

| Interaction | Why Undetected |
|-------------|----------------|
| Cross-domain artifact writes | No write-time domain check exists |
| Lifecycle state violations | Only detected during health report generation, not at mutation time |
| Transient artifact boundary crossing | `canonical_boundary.py` `enforce_boundary()` must be explicitly called — no automatic detection |
| Registry modification without lifecycle transition | No pre-write hook on `artifact_registry.yaml` |

## 5. Implicit Authority Detection

### Where Authority Exists Without Explicit Declaration

| Authority Source | Nature | Evidence |
|------------------|--------|----------|
| `engines/pipeline_orchestrator.py` | Implicit authority over all 4 chain layers | Directly calls signal, semantic, reasoning, and report engines without domain boundary checks |
| `engines/engine_runner.py` | Implicit authority to bypass chain | `run_all_engines()` produces outputs that skip SEMANTICS and REASONING layers |
| `.domainization/hooks/pre-commit` | Implicit authority to validate all domains | Runs all 5 observers across all domains without domain-scoped filtering |
| `governance/sunset_governance.py` | Implicit read authority over entire registry | Loads and scans all 132 artifacts regardless of domain |
| `.domainization/src/validation_orchestrator.py` | Implicit authority to scan all directories | `_get_trackable_files_for_enforcement()` scans `runtime/`, `governance/`, `tests/` |

### Authority Chain (Declared vs Actual)

**Declared Authority Chain** (from `domain_registry.yaml`):
```
SIGNALS (L1) → SEMANTICS (L2) → REASONING (L3) → REPORT (L4)
```

**Actual Authority Chain** (from code analysis):
```
pipeline_orchestrator.py → engine_runner.py → all engines (no domain boundary)
pipeline_orchestrator.py → semantic_engine.py → allocation only
pipeline_orchestrator.py → reasoning (synthetic objects from semantic states)
pipeline_orchestrator.py → report_engine (from engine_runner output)
```

**Gap**: The declared 4-layer authority chain is partially implemented. Only the allocation signal has full chain coverage. The remaining 13 categories flow through a synthetic placeholder path.

## 6. Canonical Boundary Risks

### Hardcoded Classification Sets

Source: `governance/canonical_boundary.py`

**Canonical Artifacts** (6 items):
```python
CANONICAL_ARTIFACTS: frozenset[str] = frozenset({
    "semantic_state_snapshot",
    "reasoning_object",
    "daily_report",
    "deployment_matrix",
    "run_context",
    "provenance_metadata",
})
```

**Transient Artifacts** (4 items):
```python
TRANSIENT_ARTIFACTS: frozenset[str] = frozenset({
    "orchestration_buffer",
    "in_memory_transform",
    "pre_validation_staging",
    "intermediate_draft_reasoning",
})
```

### Risks

| Risk | Description | Impact |
|------|-------------|--------|
| No runtime discovery | New artifact types must be manually added to hardcoded frozensets | `ValueError` raised for any unknown artifact name |
| Coverage gap | 10 artifact types in `lifecycle_state_machine.yaml` vs 6+4=10 in canonical boundary — but names don't align | Type system mismatch between governance layers |
| Promotion logic gap | `enforce_boundary()` promotes transient→canonical if persisted or passed downstream, but no code calls this automatically | Promotion is opt-in, not automatic |
| No demotion path | Once promoted to canonical, no mechanism to demote back to transient | One-way ratchet without escape |
| Static vs dynamic | Registry has 132 artifacts; boundary only knows 10 names | 122 artifacts have no canonical/transient classification |

### Name Mismatch Analysis

| Lifecycle State Machine Types | Canonical Boundary Names | Alignment |
|-------------------------------|-------------------------|-----------|
| SSOT, ENGINE, REPORT_OUT, DATA_IN, DATA_OUT, RUNTIME, DASHBOARD, SNAPSHOT, CONFIG, CALIBRATION, STEERING | semantic_state_snapshot, reasoning_object, daily_report, deployment_matrix, run_context, provenance_metadata | **No alignment** — different naming systems |

**Finding**: The canonical boundary operates on artifact *instance names* (e.g., "daily_report") while the lifecycle state machine operates on artifact *type names* (e.g., "REPORT_OUT"). These are two separate classification systems with no bridging logic.
